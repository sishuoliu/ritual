# 佛教经济学桌游：可行方案（多重正反审视版）

> 综合历史设计演变、多轮批判、技术可行性分析  
> 日期：2026-01-29  
> 状态：经过正反多重审视的落地方案

## 一、核心定位与设计哲学

### 1.1 一句话定位

**《功德轮回：众生百态》是一款以佛教经济学为主题的半合作桌游，玩家在有限资源与道德抉择中寻找个人成就与集体福祉的平衡。**

### 1.2 设计哲学的正反审视

| 设计原则 | 正面理由 | 反面风险 | 平衡策略 |
|---------|---------|---------|---------|
| **极简核心规则** | 易学难精，降低教学门槛 | 可能缺乏深度，重玩性不足 | 核心简单，策略空间来自玩家互动与事件变化 |
| **半合作模式** | 共同威胁创造紧迫感，个人目标增加张力 | 完全信息下隐藏目标感弱化 | 发愿系统公开但达成条件各异，形成差异化路径 |
| **佛教主题** | 市场独特，文化深度高 | 可能被视为冒犯或过于功利化 | 聚焦哲学概念（因果、布施），避免具象化神圣元素 |
| **经济学视角** | 资源稀缺与交换机制天然契合游戏设计 | 可能将修行过于世俗化 | 资源不只是"钱"，而是福、慧、缘等多维度 |
| **配置+脚本混合** | 数据可调，复杂逻辑可控 | 需要技术背景维护 | 简单数值用YAML，复杂效果用Python函数 |

### 1.3 核心矛盾的诚实承认

**必须承认的根本张力**：

1. **竞争 vs 无我**：游戏本质是竞争（谁得分高），佛教核心是无我。我们选择"半合作+共同胜利条件"来缓解，但无法完全消解。
2. **量化 vs 不可量化**："慧"在佛教中是洞见，不可量化；游戏中必须数值化。我们称之为"慧"但实际是"修行投入的累积"。
3. **简化 vs 准确**：游戏必须简化复杂教义。我们选择"传达体验"而非"教授教义"。

**设计目标的修正表述**：

> 本游戏不宣称模拟"成佛"，而是探索普通人在世俗责任与精神追求间如何寻找平衡。胜利者是"此生活得最有成就的人"，而非"最先觉悟者"。

## 二、游戏机制核心（v3.6 基础上的精炼）

### 2.1 核心参数

| 参数 | 数值 | 设计理由 |
|------|------|---------|
| 玩家人数 | 2-4人 | 经典半合作桌游规模 |
| 游戏时长 | 45-60分钟 | 中度策略游戏标准 |
| 回合数 | 6回合 | 足够展开策略，不过长 |
| 团队胜利条件 | 劫难≤12 且 渡化≥6 | 双重压力，避免单一瓶颈 |
| 行动次数 | 每人每回合2次 | 足够决策空间，不过复杂 |

### 2.2 资源系统

```
┌──────────────────────────────────────────────────────────────┐
│                        资源结构                              │
├──────────────────────────────────────────────────────────────┤
│  财富  ──────►  生存消耗、渡化成本、布施投入                  │
│    ↓                                                        │
│  福    ◄────── 布施、渡化、持续帮助                         │
│    ↓              ↓                                        │
│  慧    ◄────── 修行、渡化                                   │
│    ↓                                                        │
│  得分  ═ 3 × √(福×慧) + 皈依加成 + 发愿奖惩                  │
├──────────────────────────────────────────────────────────────┤
│  劫难（公共）─── 事件增加、众生超时增加、护法减少             │
└──────────────────────────────────────────────────────────────┘
```

**正面**：福慧乘积开根号鼓励平衡发展，符合"福慧双修"理念  
**反面**：开根号不是心算友好的操作；可能惩罚专业化角色  
**平衡**：提供计分表，或简化为 `福 + 慧`（需门槛）的变体规则

### 2.3 四职业差异化

| 职业 | 初始资源 | 核心优势 | 核心劣势 | 主题契合 |
|------|---------|---------|---------|---------|
| 农夫 | 财5/福2/慧2 | 渡化成本低 | 慧增长慢 | 勤劳朴实 |
| 商人 | 财8/福1/慧1 | 经济效率高 | 精神起点低 | 财施功德 |
| 学者 | 财3/福1/慧4 | 慧增长快、可讲学 | 财富紧张 | 传道授业 |
| 僧侣 | 财0/福3/慧3 | 精神起点高、化缘获财 | 无劳作能力 | 出世修行 |

**正面**：四职业形成明显差异化路径，增加重玩价值  
**反面**：平衡性难调；v3.5数据显示农夫综合排名分-8偏弱  
**平衡**：v3.6已通过事件卡微调缓解；继续用回测数据驱动调整

### 2.4 皈依抉择

| 路径 | 初始调整 | 持续影响 | 计分加成 |
|------|---------|---------|---------|
| 皈依 | 福+1，慧+1 | 正面事件+50%收益 | 福慧≥15各+16 |
| 不皈依 | 财富+3 | 无 | 福慧≥20各+18 |

**正面**：两条路线均可获胜，增加策略多样性  
**反面**：可能被解读为"不皈依也能成佛"的误导  
**平衡**：游戏说明明确：皈依与否是"游戏内策略选择"，非宗教立场

### 2.5 发愿系统（每职业2张，随机1张）

**设计意图**：
- 每回合奖励提供稳定收益
- 达成条件创造个人目标
- 奖惩分差创造张力

**正面**：秘密发愿增加半合作猜疑感；达成条件差异化增加策略深度  
**反面**：困难发愿可能导致玩家ALL-IN失败后翻盘无望  
**平衡**：简单/困难发愿奖惩差距设计合理（14-20分奖励 vs 4-8分惩罚）

## 三、佛教经济学视角的机制解读

### 3.1 核心经济学概念的游戏化

| 经济学概念 | 游戏机制 | 佛教对应 |
|-----------|---------|---------|
| **稀缺性** | 有限行动次数、资源、回合 | 人生有涯 |
| **机会成本** | 选择修行则不能劳作 | 取舍之道 |
| **公共品** | 共同劫难、团队渡化目标 | 共业 |
| **个人激励** | 发愿、个人计分 | 别业 |
| **投资** | 5财富投资→每2回合+2收益 | 布施回向 |
| **保险** | 护法行动减劫难 | 护持正法 |
| **交易** | 资源可协商交换 | 因缘互助 |

### 3.2 经济学与佛教的张力平衡

| 张力点 | 经济学视角 | 佛教视角 | 游戏处理 |
|-------|-----------|---------|---------|
| 自利vs利他 | 理性人追求自身效用最大化 | 菩萨发心利益众生 | 团队胜利条件强制合作 |
| 积累vs放下 | 积累资本是理性选择 | 执著资源是障碍 | 资源递减机制（高福慧时收益降低） |
| 竞争vs共赢 | 零和或非零和博弈 | 一切众生皆可成佛 | 半合作：共同胜利+个人排名 |
| 计算vs无住 | 理性计算收益 | 无住生心 | 骰子引入随机性，模拟"无常" |

## 四、技术实现路线（正反审视）

### 4.1 规则表达：配置+脚本混合

**做法**：
```
├── config/
│   ├── resources.yaml      # 资源定义（福、慧、财富、劫难）
│   ├── roles.yaml          # 职业初始值、行动列表
│   ├── events.yaml         # 事件卡效果（简单数值）
│   ├── beings.yaml         # 众生卡成本与奖励
│   └── vows.yaml           # 发愿条件与奖惩
│
├── engine/
│   ├── game_state.py       # 游戏状态管理
│   ├── actions.py          # 行动执行逻辑
│   ├── effects.py          # 复杂效果（条件判断、多目标）
│   └── simulation.py       # 单局执行与批量回测
```

| 方面 | 正面 | 反面 | 平衡 |
|------|------|------|------|
| **可读性** | 数值一目了然 | 逻辑分散在两处 | 效果函数命名清晰，配置引用函数名 |
| **可维护性** | 改数不改代码 | 需两套知识 | 复杂效果有单元测试覆盖 |
| **LLM生成** | 生成Python比生成复杂DSL更可控 | 仍需人工审核 | LLM仅生成片段，人粘贴并测试 |

### 4.2 回测系统

**目标**：经济压力测试，验证规则自洽

| 能做 | 不能做 |
|------|--------|
| 资源是否膨胀/崩溃 | 预测真人社交体验 |
| 规则是否产生死循环 | 评估谈判/欺骗策略 |
| 版本对比（v3.5 vs v3.6） | 测试隐藏信息场景 |
| 多策略智能体对比 | 替代真人盲测 |

**智能体设计**：
- **随机型**：每次随机选择可行行动
- **贪心渡化型**：优先渡化众生
- **保守型**：优先维持劫难安全线
- **激进型**：优先个人资源积累

### 4.3 LLM辅助（DeepSeek）

**定位**：辅助工具，非全自动生成器

| 用途 | 示例 | 风险控制 |
|------|------|---------|
| 规则片段生成 | "生成一个新的事件卡效果" | 人粘贴并测试 |
| 平衡建议 | "若提高渡化门槛会怎样" | 仅作参考，数据验证 |
| 规则解释 | "为什么这版胜率低" | 结合回测数据 |
| 多候选方案 | 模糊需求时输出3个选项 | 人选择并验证 |

**不做**：
- 一次输出完整可运行配置
- 自动修正循环（易死循环）
- 处理"让游戏更有趣"等模糊需求

## 五、阶段计划与成功标准

### Phase 1：可玩、可测的核心版本（4-8周）

| 项目 | 内容 | 验收标准 |
|------|------|---------|
| 规则固化 | v3.6规则Python实现 | 单局可跑、可复现 |
| 对局日志 | (round, phase, player, action, state_delta) | 可回放、可定位bug |
| 批量模拟 | 1000局回测 | 稳定无崩溃 |
| 智能体 | 随机+贪心渡化 | 对比差异>10% |
| 调试支持 | 单局可暂停、查状态 | 能定位规则冲突 |

**成功标准**：
- v3.6跑通1000局无崩溃
- 关键指标与旧脚本差异<5%或可解释
- 失败原因分布：劫难5-25%，渡化不足75-95%

### Phase 2：分支选择与策略多样性（4周）

| 项目 | 内容 | 验收标准 |
|------|------|---------|
| 选择树抽象 | 皈依/大乘/密宗用progression module表达 | 触发正确、无flag泥潭 |
| 策略扩展 | 保守型、激进型智能体 | 策略差异显著 |
| 报告图表 | 版本对比Markdown报告 | 可支撑决策 |
| 配置化 | 易变数值抽到YAML | 改数不改代码 |

### Phase 3：LLM辅助与体验优化（4周）

| 项目 | 内容 | 验收标准 |
|------|------|---------|
| 对话界面 | Streamlit对话框接DeepSeek | 可问答 |
| 片段生成 | LLM输出新事件/行动片段 | 至少1个需求通过LLM完成 |
| 可视化 | 简单看板：选版本→跑N局→看图表 | 非必须，按时间取舍 |

### Phase 4（视情况）：引擎抽象

**触发条件**：功德轮回稳定可玩、且有第二款游戏需求

**做法**：
- 抽出流程层（阶段、行动、牌堆、回合循环）
- 抽出模拟层（单局、批量、指标、智能体接口）
- 定义最小通用schema

**不承诺**：支持Gloomhaven、TI4等复杂游戏

## 六、风险与应对

| 风险 | 严重程度 | 应对策略 |
|------|---------|---------|
| 范围蔓延（想做通用引擎） | 高 | 主产品是佛教桌游；引擎是"做完再抽" |
| LLM生成不可用 | 中 | 只做片段+建议，人审阅合并 |
| 规则复杂导致卡壳 | 中 | 复杂条件用Python，不强行YAML化 |
| 时间被引擎分散 | 中 | 每Phase优先"功德轮回可玩" |
| 规则/实现bug难定位 | 高 | Phase 1必须做调试支持 |
| 平衡性问题 | 中 | 数据驱动，小步调整 |
| 目标用户不明 | 中 | 先定位"策略桌游爱好者+对东方哲学感兴趣" |
| 文化敏感性 | 中 | 聚焦哲学概念，避免具象化神圣元素 |

## 七、正反多重审视总结

### 7.1 方案的核心优势

1. **主题独特性**：佛教经济学视角在桌游市场极为罕见
2. **机制成熟度**：v3.6已有50+次迭代和回测数据支撑
3. **文化深度**：不是披着佛教外衣的资源游戏，而是机制与主题深度结合
4. **可落地性**：明确的Phase划分、成功标准、时间估计
5. **风险可控**：承认局限性，不追求"通用引擎"

### 7.2 方案的核心风险

1. **佛学功利化**：量化福慧、竞争觉悟，可能被批评为误读
2. **目标用户窄**：佛教主题+策略深度，受众有限
3. **平衡性挑战**：四职业+皈依+发愿的组合空间大，难以全面验证
4. **技术实现复杂**：配置+脚本混合需要维护两套知识
5. **隐藏信息缺失**：完全信息模拟无法验证"半合作"的猜疑感

### 7.3 关键决策点

| 决策 | 选择 | 理由 |
|------|------|------|
| 主产品vs引擎 | 主产品优先 | 避免范围蔓延 |
| 全YAML vs 混合 | 混合 | 复杂逻辑用Python更可控 |
| LLM全自动 vs 辅助 | 辅助 | 自动校验循环易死循环 |
| 隐藏信息 | Phase 5+延后 | 技术难度极高 |
| 竞争vs纯合作 | 半合作 | 增加张力，但承认矛盾 |
| 规则复杂度 | 中度策略 | 45-60分钟，非轻度非重度 |

### 7.4 诚实的期望管理

**能达到的**：
- 一款规则自洽、数值基本平衡的佛教主题半合作桌游
- 可批量回测验证规则一致性与经济稳定性
- 模块化代码，便于维护和迭代

**需要真人测试验证的**：
- 实际游戏体验是否"有趣"
- 半合作的猜疑感是否存在
- 佛教主题是否被玩家接受
- 教学难度是否合理

**无法通过模拟验证的**：
- 谈判/社交策略
- 隐藏信息场景
- 叛徒/欺骗体验
- 文化敏感性反应

## 八、下一步行动建议

### 立即行动（本周）

1. **确认v3.6为基准版本**：统一规则表述，解决v0.6/v3.5术语差异
2. **建立模拟器目录结构**：按Phase 1的模块划分创建代码框架
3. **编写核心对局循环**：单局可跑、可复现

7. **完善智能体**：贪心渡化、保守型、激进型
